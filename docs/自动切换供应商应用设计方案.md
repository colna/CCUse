# 自动切换供应商应用设计方案

## 一、产品概述

### 1.1 产品定位

基于 cc-switch 的架构，打造一个智能的 AI 服务供应商自动切换系统，当主供应商出现故障、限流或性能下降时，自动切换到备用供应商，确保服务连续性。

### 1.2 核心价值

- **高可用性**：自动故障转移，避免服务中断
- **成本优化**：根据价格和配额智能选择供应商
- **性能保障**：实时监控响应时间，自动切换到最优节点
- **透明管理**：可视化监控面板，掌握所有供应商状态

### 1.3 目标用户

- AI 应用开发者
- 需要高可用 AI 服务的企业
- 多供应商管理的团队
- 对成本敏感的个人开发者

pub struct SwitchEngine {
    strategy: SwitchStrategy,
    retry_config: RetryConfig,
    fallback_chain: Vec<ProviderId>,
}

impl SwitchEngine {
    pub async fn execute_request(&self, req: ApiRequest) -> Result<ApiResponse> {
        // 1. 根据策略选择供应商
        // 2. 发送请求
        // 3. 如果失败，根据策略切换到下一个
        // 4. 记录切换日志
        // 5. 返回结果
    }

    fn select_provider(&self, available: &[Provider]) -> Option<&Provider> {
        match self.strategy {
            SwitchStrategy::Priority => self.select_by_priority(available),
            SwitchStrategy::FastestResponse => self.select_by_speed(available),
            SwitchStrategy::CostOptimized => self.select_by_cost(available),
            SwitchStrategy::LoadBalance => self.select_by_load(available),
            SwitchStrategy::Smart => self.select_smart(available),
        }
    }
}

#### 3.4.2 自动切换逻辑

```rust
pub async fn execute_with_fallback(
    &self,
    req: ApiRequest,
) -> Result<ApiResponse> {
    let mut attempts = 0;
    let max_attempts = self.retry_config.max_attempts;

    while attempts < max_attempts {
        let provider = self.select_provider()?;

        match provider.send_request(&req).await {
            Ok(response) => {
                self.record_success(provider.id(), &response).await;
                return Ok(response);
            }
            Err(e) if e.is_retryable() => {
                self.record_failure(provider.id(), &e).await;
                self.mark_provider_degraded(provider.id()).await;
                attempts += 1;
                tokio::time::sleep(self.retry_config.delay).await;
            }
            Err(e) => return Err(e),
        }
    }

    Err(Error::AllProvidersFailed)
}
```

## 五、技术难点与解决方案

### 5.1 流式响应处理

**问题**：AI 服务通常使用 SSE 流式返回，切换时需要保持流的连续性

**解决方案**：

- 在代理层实现流式转发
- 切换时缓存已接收的部分响应
- 使用 tokio Stream 处理异步流

### 5.2 状态一致性

**问题**：多个供应商状态需要实时同步到前端

**解决方案**：

- 使用 Tauri 事件系统推送状态变更
- 前端使用 TanStack Query 管理状态缓存
- WebSocket 或轮询作为备选方案

### 5.3 并发请求处理

**问题**：多个请求同时到达时的供应商选择

**解决方案**：

- 使用 Arc + Mutex 保护共享状态
- 实现请求队列和限流
- 负载均衡算法分散请求

### 5.4 配额管理

**问题**：不同供应商的配额计算方式不同

**解决方案**：

- 统一配额抽象接口
- 定期查询剩余配额
- 预估请求消耗，提前切换

## 八、总结

本方案基于 cc-switch 的成熟架构，专注于解决 AI 服务高可用性问题。通过智能的自动切换机制、完善的监控体系和灵活的策略配置，为用户提供稳定可靠的 AI 服务接入方案。
