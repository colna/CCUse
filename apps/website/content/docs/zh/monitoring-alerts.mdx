# 监控与告警

仪表盘是本地代理的运行视图。用它确认请求是否正常流动、供应商是否健康，以及故障转移事件是否符合当前切换策略。

## 状态卡片

顶部状态卡片会每隔数秒刷新一次，汇总最近 24 小时窗口：

| 卡片         | 含义                                  |
| ------------ | ------------------------------------- |
| 当前供应商   | 当前健康快照中的第一个 healthy 供应商 |
| 今日请求     | 24 小时指标时间桶内的请求总数         |
| 成功率       | 最近请求时间桶的加权成功比例          |
| 平均响应时间 | 最近请求时间桶的加权平均延迟          |

成功率低于 `95%` 时应视为警告。常见原因是某个供应商失败、被限流，或上游间歇性返回错误。

## 成功率

成功率折线展示最近 24 小时的 5 分钟时间桶。每个点来自请求日志：

```txt
success_rate = successful_requests / total_requests
```

可以用它发现：

- 修改策略或供应商配置后，成功率突然下降。
- 每天固定时间段出现重复失败。
- 禁用异常供应商后，成功率是否恢复。

如果图表为空，说明当前 24 小时窗口里还没有请求日志。

## 响应时间

响应时间折线展示每个 5 分钟时间桶的平均延迟和 P95 延迟。

解读方式：

- 平均值代表常规请求速度。
- P95 代表较慢的尾部请求。
- P95 上升但平均值稳定，通常表示少量上游请求卡顿。
- 两条线一起上升，通常表示当前供应商或网络路径整体变慢。

对于 Cursor 这类交互客户端，优先关注 P95。即使平均值看起来正常，少数慢请求也会明显影响输入体验。

## 供应商成本

成本饼图按供应商汇总最近 24 小时 token 使用量，并乘以每个供应商配置的 `cost_per_1k_tokens`。

用它确认：

- 成本优先或智能策略是否真的把支出转向更便宜的供应商。
- 是否有某个供应商承担了超过预期的流量。
- 缺失成本值是否让支出显示为 `0`。

依赖成本报表前，请为所有可比较的供应商填写成本元数据。

## 切换时间线

切换时间线就是最近的切换历史查询。它按时间倒序读取 `switch_history` 最新 50 条记录。

每一行包含：

| 字段       | 含义                              |
| ---------- | --------------------------------- |
| 时间       | 切换事件记录时间                  |
| 原供应商   | 失败或被替换的供应商              |
| 目标供应商 | 成功尝试中选中的供应商            |
| 策略       | 该请求使用的策略快照              |
| 原因       | 机器可读原因，例如 `upstream_502` |
| ID         | 用于支持与调试的数据库行 id       |

展开某一行可以查看原因和 ID。排查用户反馈时，把这条记录的时间戳与客户端请求时间、仪表盘显示的供应商对上。

## 告警信号

CCUse 在两个地方暴露告警信号：

- 供应商健康状态变化会发送 `provider-status-changed`，让前端立即刷新供应商状态。
- 桌面通知使用系统原生通知中心，用于故障、配额、切换提示等通知动作。

如果没有明显告警，先看仪表盘。供应商列表健康、成功率稳定、切换时间线为空，通常说明当前没有需要通知的异常。

## 排查清单

当请求失败或延迟升高时：

1. 检查当前供应商和供应商状态。
2. 打开切换时间线，查看最新原因。
3. 对照相同时间点的成功率和响应时间。
4. 如果流量切到意外供应商，检查供应商成本图。
5. 确认客户端仍使用复制的 Base URL 和 `sk-local-...` key。

如果持续出现 `rate_limited` 或 `upstream_5xx`，可以添加另一个启用供应商，或切换到 `load_balance` / `smart`，让流量分散到更多容量。
