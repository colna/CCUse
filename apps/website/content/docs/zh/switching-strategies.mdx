# 切换策略

CCUse 会在每个请求开始时选择一个供应商。如果被选中的供应商出现可重试的网络错误、上游错误、限流或解码错误，切换引擎会在当前重试链路中排除这个供应商，并尝试下一个候选项。

流式响应有单独规则：一旦 stream 已经开始，CCUse 不会在流中途切换供应商。客户端应重试请求，下一次请求开始时可以切到另一个供应商。

## 快速选择

大多数场景优先使用 `smart`。它会在可用性、延迟、价格和手动优先级之间做平衡，不需要你频繁调参。

如果目标很明确，再选择更窄的策略：

| 策略     | 最适合                       | 主要取舍                             |
| -------- | ---------------------------- | ------------------------------------ |
| 优先级   | 稳定的主备路由               | 备用供应商通常只在主供应商失败时使用 |
| 最快响应 | 编程助手、聊天、低延迟交互   | 可能选到更贵的供应商                 |
| 成本优先 | 批处理、后台任务、预算控制   | 可能选择更慢的供应商                 |
| 负载均衡 | 把流量分散到多个可用 key     | 效果依赖准确的优先级权重             |
| 智能策略 | 混合约束下的日常默认生产策略 | 需要较合理的延迟和成本元数据         |

## 优先级

`priority` 会选择优先级数值最低的已启用供应商。优先级 `10` 会比优先级 `50` 更先被选中。

适合在这些场景使用：

- 你有一个稳定主供应商，以及一个或多个备用供应商。
- 你希望路由结果可预测，方便审计或账单归因。
- 你正在测试新供应商，只希望主供应商失败后才让它接流量。

建议优先级数值留出间隔，例如 `10`、`20`、`30`，后续插入新供应商时不必全部重排。

## 最快响应

`fastest` 会选择滚动响应时间最低的供应商。CCUse 会在正常请求中记录延迟，并用这个移动信号服务后续请求。

适合在这些场景使用：

- 用户体验高度依赖首字响应速度。
- 多个供应商的质量和成本接近。
- 你主要代理 Cursor 或 Claude Desktop 一类对体感延迟敏感的客户端。

如果还没有延迟历史，CCUse 会退回到已启用供应商顺序，也就是按优先级排序后的顺序。

## 成本优先

`cost` 会选择配置了最低每 token 成本的供应商。没有成本元数据的供应商不会参与成本比较；如果所有供应商都没有成本数据，CCUse 会退回到优先级顺序。

适合在这些场景使用：

- 你运行大量后台 prompt 或批处理工作流。
- 多个供应商的输出质量接近。
- 你想用简单默认策略控制预算，不想编写自定义路由规则。

如果要依赖成本策略，请为可比较的供应商都填写 `cost_per_1k_tokens`。空字段会降低成本路由的有效性。

## 负载均衡

`load_balance` 会在已启用供应商之间做加权 round-robin。权重来自优先级：优先级数值越低，获得的请求槽越多。

适合在这些场景使用：

- 你有多个健康的同模型供应商或多个 key。
- 你想降低单个上游账号触发限流的概率。
- 你希望分散流量，同时仍让更偏好的供应商承接更多请求。

如果想接近均匀分配，可以使用接近的优先级，例如 `10`、`10`、`20`。如果希望某个供应商承接大多数请求，就拉大优先级差距。

## 智能策略

`smart` 会按四个维度为每个供应商打分：

| 维度     | 默认权重 | 含义                              |
| -------- | -------- | --------------------------------- |
| 健康度   | `40`     | 优先选择当前可用的供应商          |
| 响应时间 | `30`     | 优先选择滚动延迟更低的供应商      |
| 成本     | `20`     | 优先选择每 token 成本更低的供应商 |
| 优先级   | `10`     | 保留你的手动供应商偏好            |

适合在这些场景使用：

- 你需要日常使用里最稳妥的默认策略。
- 供应商在可靠性、速度、成本上各有优劣。
- 你希望自动切换，但不想频繁手动调整供应商顺序。

交互型场景可以提高响应时间权重；预算敏感场景可以提高成本权重；如果仍希望手动偏好影响结果，保留一部分优先级权重。

## 失败处理

所有策略共享同一套重试行为：

- 已禁用供应商不会被选中。
- 可重试的供应商失败会在当前流程中把该供应商标记为 degraded。
- CCUse 会继续尝试其他已启用供应商，直到达到重试上限。
- 非可重试错误，例如请求结构无效，会直接返回给客户端。

最稳妥的配置方式是：至少启用两个供应商，给它们明确优先级；使用成本相关策略时填写成本元数据；每次切换策略后，用一个短请求验证本地端点。
